# CLAUDE.md — project memory

Hebrew/RTL credit-card & bank analyzer ("מנתח עסקאות"). This file is auto-loaded
each session so context carries over. **Never put secrets here** (it's committed).

## Architecture
- **frontend/** — React 19 + TypeScript + Vite + Tailwind v4, Recharts, Framer Motion, React Router 7. RTL/Hebrew.
- **backend/** — FastAPI (Python 3.11), Pandas. Sessions are an **in-memory** `sessions: dict[str, pd.DataFrame]` (wiped on Render cold-start; the frontend re-restores from Supabase).
- **bank-sync/** — a **local** Node tool the user runs on their own Mac. Scrapes Israeli banks (`israeli-bank-scrapers`), writes a transaction snapshot to the user's Supabase. NOT deployed (Docker image copies only frontend/ + backend/).
- **Supabase** — `saved_transactions` (JSONB snapshots) + `user_category_rules`; RLS; new-style `sb_publishable_…` anon key. bank-sync and the dashboard both sign in **as the user** with the anon key — **never `service_role`**.

## Deploy
- Render free tier, single Docker service, **auto-deploys from `main`**.
- Backend/frontend changes go live only when merged to `main`. bank-sync changes are pulled locally by the user.
- Workflow: develop on the session's `claude/…` branch (currently **`claude/dashboard-update-a43etv`**), then fast-forward `main` and push to deploy.
- `ANTHROPIC_API_KEY` is set on Render (enables the AI categorizer).

## Security rules (do not violate)
- **Never** ask for / accept the user's bank passwords, Supabase login, or sync token. They live only in the OS keychain.
- Never use the Supabase `service_role` key.
- `.gitignore` blocks `.env`, secrets, and `debug/`. Don't commit secrets.

## Domain decisions (locked)
- **Per-person owner attribution** (`_owner`): מור / שלי / משותף(joint). A *personal* account forces its owner; a *joint* account splits by salary keyword (`בנק פועלים משכורת`→מור, `ישראכרט מש משכורת`→שלי). **בנק דיסקונט must be joint (משותף).**
- **Accounts**: discount (joint bank), MAX (מור), Isracard `isracard`=מור, Isracard `isracard-2`=שלי.
- **Income-month shift**: end-of-month salaries (day ≥25) attributed to the *next* month. bank-sync records this in the `חודש` / `חודש_חיוב` columns. **All month grouping must go through `_month_series()` in routes.py** — never re-derive the month from the raw date (that discards the shift).
- **Taxonomy (2026-07, Shelly's tree — LOCKED)**: 13 real-life categories — טיסות ותיירות, הוצאות שוטפות (שכ"ד/ארנונה/חשמל/מים/גז/אינטרנט/סלולר/סטרימינג/דלק/ביטוח/מנויים), תרופות וטיפולים, **פארם** (רשתות פארם; תת-קטגוריות תרופות/טיפוח נקבעות **פר-עסקה** עם הכפתור, לכן פארם מוחרגת מהסריקה האוטומטית — `AI_SUBCATEGORIZE_SKIP`), בילויים (מסעדות/קפה/מזון מהיר/סרטים/מופעים/אטרקציות/פיס), אוכל (**קניות גדולות**=רשתות מותג, **סופרים קטנים**=מכולות/מינימרקטים/פרימדונה + משלוחים/מאפיות/קצביות/אלכוהול/שוברי מזון), קניות (אופנה/אלקטרוניקה/ריהוט/דברים לבית/אונליין/תכשיטים), טיפוח, חוגים וספורט (כולל לימודים), אירועים ומתנות, טכנולוגיה (**AI ירדה לתת-קטגוריה** — תת `AI` תחת טכנולוגיה, אייקון 🤖; `AI_CATEGORY='טכנולוגיה'`+`AI_SUBCATEGORY='AI'`), הוצאות משתנות (כל התחבורה: מוניות/חניונים/אגרה/תח"צ/טיפולים רכב), **סושי** 🐾 (חיית המחמד: וטרינרית/אוכל וחטיפים) + טכניות: העברת כספים, העברה להשקעות, משיכת מזומן, הוראות קבע, שונות.
- **Old→new migration**: `CATEGORY_MIGRATION`/`CATEGORY_PAIR_MIGRATION`/`migrate_category` in constants.py (mirrored in frontend `constants.ts` `migrateRule` + bank-sync `migrateCategory`). Applied FIRST in restore (before hygiene — old snapshot rows are translated, never nuked to שונות), to incoming rules/pins on the fly, by `Layout.tsx` at login (rewrites the stored Supabase rules once), and by bank-sync `refreshMiscCategories` (retag). Old names must NEVER be re-added to `CATEGORY_ICONS`.
- **Dynamic categories**: table `user_categories` (§8 in supabase_setup.sql) — a category name typed in the edit drawer that isn't in the tree is auto-persisted there, loaded at login, passed as `custom_categories` to `/restore-session`, and honored everywhere validity is checked (hygiene, rules, pins, `/merchants/category`; per-session registry `SESSION_CUSTOM_CATS`).
- **Categorization**: ~1000-keyword catalog. Source of truth = `backend/app/core/constants.py`; `bank-sync/src/categorize.js` is **GENERATED — run `cd backend && python scripts/generate_bank_sync_categorize.py` after any constants.py change** (never edit the JS by hand). Foreign card transactions (trailing 2-letter country code, no Hebrew, not `IL`) → `טיסות ותיירות` — **except** merchants in `FOREIGN_EXEMPT_KEYWORDS` (Netflix/Spotify/PayPal…: online services billed from abroad, not trip spend). Leftover `שונות` → AI fallback via **`POST /ai-categorize`, fired by the frontend in the background AFTER restore returns** (never inline — it blocked first paint); results returned as `ai_categorized` and persisted as user rules. The fallback is **two-phase** (`ai_categorizer.py`): phase 1 classifies only merchants Claude recognizes with certainty (guessing from the name is forbidden); unknowns go to phase 2 where a **web search per merchant is mandatory** — answers produced without searching are discarded and rows stay `שונות`. Installment suffixes (`(תשלום n/N)`) are stripped before querying so all installments resolve together.
- **Rule hygiene**: `'אחר'` is a chart-legend bucket, NOT a category. Rules whose category isn't in `CATEGORY_ICONS` are ignored by restore, deleted from Supabase by the frontend on login, and skipped by bank-sync `applyRules`; snapshot rows stored with an invalid category are reset to `שונות` and re-categorized (restore + `npm run retag`). Category pickers must offer `ASSIGNABLE_CATEGORIES` (no `אחר`).
- **Unconditional overrides** (Psagot→`העברה להשקעות`, foreign-card→travel, AI tools→`טכנולוגיה`+תת `AI`) live in `data_processor.apply_unconditional_overrides()` — shared by `process_data` and `restore_session` so they never drift, and run on **all** rows. `AI_OVERRIDE_KEYWORDS` is the curated AI-tool list. The AI-tool override **beats user rules** (`apply_ai_tool_override` re-runs after rules and sets BOTH category and subcategory; mirrored in bank-sync `applyRules`, which protects AI merchants by DESCRIPTION — not by category, since טכנולוגיה also holds non-AI merchants).
- **Subcategories** (`קטגוריה_משנה`): keyword-seeded via `SUBCATEGORY_KEYWORDS` (scoped per parent) in `data_processor.derive_subcategory()` (runs **after** user rules; **seeded keywords WIN over stored/AI/rule subcategories** — same supremacy as the category catalog — and fill empties elsewhere; **locked/pinned rows are never touched**). Seeded parents: the full 2026-07 tree (see the taxonomy bullet). **bank-sync's whole categorize.js is GENERATED** (`backend/scripts/generate_bank_sync_categorize.py`). Manual assignment persisted as `user_category_rules.subcategory`. Catalog exposed via `GET /api/categories/catalog` (includes seedless pickables: שופינג, בילויים עם חברים, פארם's תרופות/טיפוח). **AI subcategory split is fully automatic**: `POST /ai-subcategorize-all` sweeps every category with unsubcategorized rows **except `AI_SUBCATEGORIZE_SKIP` = {שונות, פארם}**, fired by `Layout.tsx` in the background **chained after `/ai-categorize`** on every restore — zero user action. Both group by canonical key and let Claude reuse existing names or **create new subcategories** via the same two-phase no-guessing flow as categorization (phase 1 = certain only; phase 2 = web search mandatory, unsearched answers discarded), with a per-(category, merchant) in-process cache incl. misses; manual assignments are never clobbered; assignments are persisted as merchant rules. `/ai-categorize` also passes each merchant's `ענף_מקור` to Claude as a hint. `קניות גדולות` is keyword-seeded FIRST under אוכל (so `יינות ביתן` isn't stolen by the `יין` keyword of אלכוהול ומשקאות).
- **Month-by-month category comparison**: `GET /api/charts/v2/category-monthly-comparison` (shift-aware `_month_series`, `date_type`), each cell carries amount + `pct_of_month` + MoM delta.
- **Catalog wins when it knows** (restore + `npm run retag` + bank-sync `applyRules`): the keyword catalog is re-run over **expense** rows too (not just `שונות`) and beats BOTH stale stored categories AND conflicting rules — a rule contradicting a catalog hit is a stale AI guess and its *category* part is ignored (the *subcategory* part of a rule always applies). Precedence: **unconditional overrides > catalog > rules > issuer (`ענף_מקור`) > stored category**. Catalog-unknown rows keep stored/rule categories; income rows are never second-guessed. Consequence: manual category edits only stick for catalog-unknown merchants.
- **Trip-window heuristic** (restore `apply_trip_window_heuristic` + bank-sync `refreshMiscCategories`): latin-only, non-exempt `שונות` rows within ±3 days of CONFIRMED overseas spend (latin + trailing country suffix, already travel) → `טיסות ותיירות` — catches truncated-suffix trip rows ("CHARM BKK"). Hebrew rows and far-away dates are never touched; Israeli flight bookings (Hebrew descriptors) are not anchors. AI phase-2 prompt also teaches: business located abroad = travel, שונות only with zero signal.
- **Card-bill payments** (bank-row outbound matching card keywords) are stripped to avoid double-counting. **Installments** are split per billing month. `שונות` should be rare (genuine one-offs only).
- **Issuer category (`ענף_מקור`)**: the card company's own sector per transaction — MAX sends it free on every sync; Isracard needs opt-in `ISRACARD_EXTRA_INFO=true` (extra request per txn, anti-bot risk). bank-sync stores it on every row (`mergeSnapshots` backfills history). It's a **weak** signal: fills only what the keyword catalog left in `שונות`, before user rules and the AI fallback. Mapping = `ISSUER_CATEGORY_RULES`/`map_issuer_category` in `constants.py`, mirrored as `categoryFromIssuer` in `bank-sync/src/categorize.js` — keep identical.
- **Canonical merchant key**: rules are matched via `normalize_merchant` (backend) / `normalizeMerchant` (bank-sync, identical): strips installment suffix + `PAYPAL*/PP*/GOOGLE*/FACEBK*` prefixes, collapses whitespace, lowercases. A rule saved from one descriptor variant hits all variants (all installments!). bank-sync rule maps must be built with `buildRuleMap`.
- **Single-transaction pins ("אל תשנה סיווג של עסקאות דומות")**: the edit drawer has a checkbox that scopes a category/subcategory edit to ONE transaction — no merchant rule is created, similar transactions are untouched (each ביט transfer can carry its own category). Persisted in Supabase `transaction_overrides` keyed by a stable fingerprint `txn_fingerprint(date|amount|desc)` (`data_processor.py`; the restore dedup triple, so unique per session and stable across restores — **run the idempotent section 7 of `supabase_setup.sql` once in Supabase**). Applied LAST on `/restore-session` → **highest precedence of all** (beats unconditional overrides, catalog, rules, AI) and sets `_locked` on the row; `derive_subcategory`, `/merchants/category`, `/ai-categorize`, `/ai-subcategorize*` and `/ai-audit` all skip locked rows (`locked_mask`). A normal (unchecked) edit explicitly unpins the row and deletes its override. bank-sync is deliberately unaware — pins live only in the restore path.
- **Edit flow (2026-07, Shelly's UX)**: an UNCHECKED category/subcategory edit applies to **all similar transactions immediately** in the live session (canonical-merchant match, pinned rows skipped) — not only on the next restore. The drawer editor **stages** category+subcategory+note and saves them together (one שמור); picking a category switches the subcategory chips to that parent's list (`subcategoryCatalog` prop). **Notes** (הערות) persist in Supabase `transaction_notes` (§9, fingerprint-keyed) and are re-applied on `/restore-session` via `transaction_notes` in the body. **Custom subcategories stay pickable**: `GET /api/categories/catalog?sessionId=…` merges every in-use subcategory per parent, and the drawer remembers newly typed names (localStorage + `addCustomSubcategory`).
- **Longest keyword wins**: `KEYWORD_TO_CATEGORY`/`KEYWORD_ENTRIES` are sorted longest-first (stable), so the most specific keyword beats a shorter generic one across categories (`רמי לוי תקשורת` → תקשורת, not מזון). Catalog declaration order no longer decides cross-category collisions.
- **AI category audit = exhaustive web verification**: `POST /ai-audit` skips catalog-known merchants server-side (the catalog governs them) and web-verifies the rest merchant-by-merchant — `audit_merchants` runs small batches with MANDATORY web search (unsearched responses discarded, no searchless fallback), verdicts cached per (merchant, current). Stage 3 of the `Layout.tsx` background chain loops slice-after-slice (exclude list) until `remaining == 0`, auto-applies proposals with `confidence ≥ 0.85`, and **pins every web-CONFIRMED merchant as a rule** (`verified` list in the response) so each merchant is verified once, ever. Progress reported via `AI_PROGRESS` stage `auditing`.
- **AI progress meter**: the background chain reports live progress into `AI_PROGRESS` (in-memory, keyed by session); `GET /ai-progress` returns `{stage: idle|categorizing|categorized|subcategorizing|done, done, total, detail}`. `Layout.tsx` polls it every 2s while the chain runs and shows a floating RTL pill (bottom-right), ending with "הקיטלוג הסתיים ✓".

## bank-sync commands (run locally)
- `npm run setup` — manage account registry + creds (keychain). Shows each account's current owner; can update a login.
- `npm run sync` — scrape + **merge** (never wipes). `ACCOUNTS=isracard-2 npm run sync` limits to specific account keys.
- `npm run resync` — full re-pull (`--fresh`). Now **preserves** rows for accounts that fail mid-run; keeps last 3 snapshots as backups.
- `npm run retag` — re-attribute `_owner` AND re-run the keyword catalog on `שונות` rows in the stored snapshot, **no bank login** (use after owner changes, catalog updates via `git pull`, or while a provider is blocked). Needed because `mergeSnapshots` never overwrites stored fields — new catalog keywords don't reach old rows via `npm run sync`.
- `npm run check` — read-only sanity report (accounts, per-owner split, uncategorized `שונות`, sign/outlier checks, and pipeline **accuracy vs `golden.json`** when present).
- `npm run golden` — seed/refresh `bank-sync/golden.json` (gitignored) with top-spend merchants labeled by their current category; user edits = ground truth, never overwritten. `npm run check` then reports keyword+issuer pipeline accuracy against it.
- Isracard anti-bot ("Block Automation" / HTTP 429): run headful (`SHOW_BROWSER=true` in `.env`), space logins out, and pull the two Isracard cards in **separate** runs (delays via `ACCOUNT_DELAY_MS` / `SAME_PROVIDER_DELAY_MS`).
